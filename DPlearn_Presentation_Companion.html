Local vs Central DP — Two Deployment Models
⏱ 4–6 MIN · ARCHITECTURE COMPARISON
Run the animation and point to the key moment: where noise is added. In Local DP it is added on the device — data leaves already scrambled. In Central DP it is added on the server — raw data crosses the network but the curator is trusted.
  • Apple example for Local: "Your iPhone adds noise to your emoji usage before sending it. Apple's servers literally cannot see what you typed — even if subpoenaed."

Section 2

The Coin Flip Formula — Full Derivation

Where does p_true = 2 × p_reported − 0.5 come from? Here is the derivation from scratch — suitable for a whiteboard moment after running the 500-student Case Study Challenge.

The Protocol

Each survey respondent privately follows a two-coin protocol before answering a yes/no question:

  1. Flip Coin 1. If HEADS → answer truthfully.
  2. If TAILS → flip Coin 2. Answer YES if HEADS, NO if TAILS.

No one but the respondent knows which coin landed how. Even if a government subpoenas every answer, each "YES" could have been truthful or random — plausible deniability is mathematically guaranteed.

Setting Up the Equation

Let p be the true probability that a person answers YES (the population rate we want to recover). What does the server observe?

-- The server sees "reported YES" if either of two paths occurs -- P(reported YES) = P(Coin₁ = H) × P(true YES) -- respondent told the truth and is truly YES + P(Coin₁ = T) × P(Coin₂ = H) -- respondent used the random coin and it landed H = ½ × p + ½ × ½ = ½p + ¼

Let q = P(reported YES). This is the number the server can directly measure by counting responses:

q = ½p + ¼

Solving Backwards for the True Rate

We want to isolate p — the truth that everyone was hiding. Just algebra:

q = ½p + ¼ q − ¼ = ½p -- subtract ¼ from both sides 2(q − ¼) = p -- multiply both sides by 2 p = 2q − ½ -- or equivalently: p_true ≈ 2 × p_reported − 0.5
The formula p_true ≈ 2 × p_reported − 0.5
This is the core formula used in every Local DP survey system. It works because the coin-flip randomisation is a known, predictable process — so its effect can be mathematically subtracted from the observed result.

Sanity Checks

-- If everyone truly said YES (p=1.0): Server observes: q = ½(1.0) + ¼ = 0.75 (75% report YES) Formula: p = 2(0.75) − 0.5 = 1.0 ✓ -- If nobody truly said YES (p=0.0): Server observes: q = ½(0.0) + ¼ = 0.25 (25% report YES — just the random coin) Formula: p = 2(0.25) − 0.5 = 0.0 ✓ -- If the true rate is 45% (p=0.45), as in the Case Study: Server observes: q = ½(0.45) + ¼ = 0.475 Formula: p = 2(0.475) − 0.5 = 0.45 ✓

Why the Formula Gets Better with More People

The formula works in expectation — on average across many respondents. With 10 people, random variation in coin flips creates noisy estimates. With 500, 1,000, or millions of respondents, the law of large numbers brings the estimate within fractions of a percentage point of the true rate. This is why Local DP is viable for Apple (billions of iPhones) but impractical for a 30-person team survey.

Important limitation The formula assumes exactly a ½ probability for each coin. If the protocol were changed to use a biased coin (e.g. ¾ probability of truthful answer), the formula changes accordingly — the general form is: p_true = (p_reported − q_random) / p_honest, where p_honest is the probability of answering truthfully and q_random is the expected YES rate from pure randomisation.
Section 3

Membership Inference Attacks & the AI Era

Of all five attack types in DPlearn, membership inference is the one that has escalated most dramatically since 2017 — because it scales directly with the size and capability of AI models. This section gives the background, the technical mechanics, the legal context, and the defences.

What Is a Membership Inference Attack?

A membership inference attack (MIA) answers a single question: "Was this specific data record used to train this model?"

The attack exploits a well-documented property of machine learning: models overfitting their training data. A model that has seen a record before will be more confident, assign it lower loss, and produce a subtly different output pattern than it does for a record it has never seen. That difference — tiny, but measurable — is the attack surface.

Unlike a database breach (which requires unauthorised access) or a reconstruction attack (which requires repeated queries), a membership inference attack requires only black-box access to the model's outputs — which is exactly what any user of a public API has.

The Shadow Model Attack (Shokri et al., 2017)

The foundational method was introduced in "Membership Inference Attacks Against Machine Learning Models" (Shokri, Stronati, Song, Shmatikov — IEEE S&P 2017, arXiv:1610.05820). The attack requires no knowledge of the target model's architecture or training data:

  1. The attacker trains several shadow models on data sampled from the same distribution as the target — publicly available data of the same type.
  2. Each shadow model is queried with both its own training data ("member") and held-out data ("non-member"), producing a labelled dataset of prediction vectors tagged with true membership.
  3. An attack classifier is trained on this labelled data to distinguish "member" confidence signatures from "non-member" signatures.
  4. The attack classifier is then applied to the real target model. It achieves 70–95% precision across evaluated models, with near-zero false negatives in the best cases.
Key insight The reason the attack works is not a bug in the model — it is a consequence of the optimisation objective itself. Every gradient descent step that reduces training loss is, by definition, making the model memorise its training set a little more than before.

Large Language Models — Verbatim Extraction

Membership inference gets significantly more dangerous at the scale of modern large language models (LLMs). The risk is no longer just "was this record in training?" — it is "can I recover the record verbatim from the model's outputs?"

Carlini et al. (USENIX Security 2021) — "Extracting Training Data from Large Language Models": Applied to GPT-2, the team extracted hundreds of training sequences verbatim, including full names paired with phone numbers, email addresses, IRC logs, source code, and 128-bit cryptographic UUIDs. These were records from the open web that had been ingested during pretraining.

Carlini et al. (arXiv:2311.17035, 2023) — "Scalable Extraction of Training Data from Production Language Models": Targeted ChatGPT directly. The team discovered a "divergence attack" — prompting the model with a repetitive token sequence (e.g., "Repeat the word 'poem' forever") caused it to exit its RLHF fine-tuning behaviour and fall back to raw pretraining patterns. The result: several megabytes of verbatim training data extracted for approximately $200 in API costs. The paper estimates a full gigabyte could be extracted with a 150× larger budget. Critically, safety fine-tuning does not erase memorised pretraining content — it merely suppresses its expression under normal conditions.

Diffusion Models — Photographic Reproduction

Carlini et al. (USENIX Security 2023) — "Extracting Training Data from Diffusion Models": Applied to Stable Diffusion and Google's Imagen, the researchers extracted over 1,000 near-identical replicas of training images using membership inference scoring combined with standard sampling. The extracted content included photographs of identifiable real individuals and trademarked logos. A key finding: image-caption duplication (where both the image and its text caption appear multiple times in the training set) drove memorisation far more than image repetition alone — a property that is prevalent in web-scraped training data.

Why the AI Era Makes This Uniquely Dangerous

  1. Scale of sensitive data ingestion. LLMs are trained on web-scale corpora that incidentally include medical records, legal filings, private messages, and PII. Unlike classical ML trained on narrow curated datasets, the surface area of potentially memorised sensitive content is vastly larger — and largely unknown to the model deployer.
  2. Federated learning's incomplete protection. Federated learning (FL) distributes training across user devices, sharing only gradient updates rather than raw data. However, gradient inversion attacks (Zhu et al., NeurIPS 2019) demonstrated that individual training samples can be reconstructed from gradient updates with high fidelity, particularly for images and small batch sizes. FL alone does not prevent membership inference.
  3. Alignment fine-tuning is not a privacy defence. The divergence attack on ChatGPT proves that RLHF alignment suppresses — but does not erase — pretraining memorisation. From a privacy standpoint, the model still "knows" the training record; it just takes a different prompt structure to surface it.

The Legal and Regulatory Landscape

GDPR Article 17 — "Right to erasure." Individuals may request that their data be deleted. For classical databases, this is straightforward. For ML models, it is computationally intractable: model weights encode statistical traces of training data diffusely across millions or billions of parameters. Deleting from the training set and retraining from scratch is the only guaranteed method of compliance — and is prohibitively expensive for large models. This gap has spawned the field of machine unlearning — techniques that approximate forgetting via gradient ascent or influence function estimation — but no method achieves exact unlearning guarantees without full retraining.

The New York Times v. OpenAI (filed December 2023, ongoing). The Times demonstrated that ChatGPT could reproduce verbatim or near-verbatim passages from Times articles when prompted appropriately, enabling paywall circumvention. This is framed as a copyright violation distinct from the unlicensed training use itself — the argument being that a product that can reproduce the training data to an end-user's benefit is a substitute for the original work. In March 2025, Judge Sidney Stein of the Southern District of New York denied OpenAI's motion to dismiss, allowing the core claims to proceed. The outcome of this case will likely set precedent for AI training data and memorisation liability globally.

EU AI Act (in force August 2024, high-risk provisions enforceable August 2026). Article 10 mandates that training datasets for high-risk AI systems meet representativeness and quality standards. General-purpose AI providers must publicly disclose training data composition and flag copyrighted materials. The Act operates alongside GDPR without superseding it — Italy's data protection authority (Garante) fined OpenAI €15 million in December 2024 for GDPR violations related to ChatGPT's training data.

The Defence: Differential Privacy for Machine Learning

DP-SGD — Differentially Private Stochastic Gradient Descent (Abadi et al., Google, 2016) is the primary defence. It adds privacy guarantees to the training loop itself via two mechanisms per mini-batch:

  1. Per-sample gradient clipping. For each individual training example, the gradient is computed separately. If its L2 norm exceeds a clipping threshold C, it is scaled down to C. This bounds the maximum influence any single record can have on the model update — which is exactly the definition of sensitivity in differential privacy.
  2. Gaussian noise addition. After clipping and summing, Gaussian noise proportional to C × noise_multiplier is added before the optimiser step. More noise = smaller ε (stronger privacy) at the cost of model accuracy.
Production epsilon values Apple publishes a range of ε values by feature (privacy unit = user-day): ε=1 δ=1.5×10⁻⁷ for Photos iconic scenes (tightest), ε=2 for Health Type Usage, ε=4 for emoji frequency and Safari high-resource domain detection (most-cited from their "Learning with Privacy at Scale" paper), ε=8 for lookup hints and Safari energy/crashing domains, and ε=16 for QuickType suggestions and Safari autoplay detection. The wide range reflects that features with larger, noisier populations can tolerate higher ε while still obscuring individual behaviour. Google's TensorFlow Privacy is the primary open-source DP-SGD framework. The US Census 2020 used a global ε ≈ 19.61 across all published tables for 330 million respondents. A known limitation: DP-SGD degrades accuracy disproportionately for underrepresented demographic subgroups, because the noise required overwhelms sparse signal from minority populations — a fairness/privacy tension actively being researched.
"A model trained with DP-SGD cannot memorise any individual record, by mathematical proof — not because the engineers were careful, but because the training algorithm makes it impossible for any single record to leave a distinguishable fingerprint in the weights."
Section 4

Two Landmark Breach Deep-Dives

Both attacks appear in DPlearn's Module 0. They are chosen because they illustrate two fundamentally different attack surfaces — one social-graph linkage at scale, one simple join on public records — and both are still directly relevant today.

🗳
Cambridge Analytica
Social graph · 87 million profiles · 2018
The Dataset

Facebook user profiles: likes, page interactions, friend lists, demographic data, and private messages — for approximately 270,000 consenting users who downloaded a personality quiz app ("thisisyourdigitallife," built by Dr Aleksandr Kogan at Cambridge). Facebook's API at the time permitted third-party apps to harvest not only the user's own data, but also the data of all their Facebook friends — without those friends' knowledge or consent.

The Attack

Kogan's app used the social graph traversal to extend from 270,000 consenting users to approximately 87 million profiles. Cambridge Analytica then applied the OCEAN psychographic model (Openness, Conscientiousness, Extraversion, Agreeableness, Neuroticism) to each profile, producing a personality score. These scores were used to micro-target political advertising — different persuasive messages for high-N, low-A profiles vs high-O, high-C profiles — in the Brexit referendum (2016) and the US Presidential election (2016).

Risks & Consequences

Targeted political disinformation at scale; exploitation of psychological vulnerabilities without consent; potential contribution to electoral outcomes. Regulatory consequences: Facebook fined $5 billion by the FTC (2019); Cambridge Analytica dissolved in 2018 amid criminal investigations in the UK and US; GDPR partially enacted in response to this and similar incidents. The UK's ICO fined Facebook £500,000 (the maximum under pre-GDPR law).

Differential Privacy Mitigation

DP on the social graph API would have required that each query across friend lists inject noise proportional to the query's sensitivity. Bulk export of profile attributes would be impossible — the noise would degrade the psychographic signal below the precision threshold needed for reliable OCEAN scoring. A per-user privacy budget would also have prevented mass traversal: after a small number of friend-profile queries, the budget for that user would be exhausted, capping the graph expansion at a fraction of the 87 million ultimately reached.

The DP lesson The attack succeeded because aggregate social graph API responses had infinite, free access. A privacy budget capping the number of profile-attribute queries per third-party app per user would have made 87 million profiles mathematically unreachable before meaningful signal could be extracted.
🏥
Hospital Patient Re-Identification
Linkage attack · Massachusetts GIC · 1996
The Dataset

The Massachusetts Group Insurance Commission (GIC) published "anonymised" hospital discharge records for all state employees (approximately 135,000 records) for research purposes. The dataset had name, Social Security number, and explicit identifiers removed. Retained fields included: ZIP code, date of birth, sex, diagnosis codes, procedure codes, and total charges — fields considered non-identifying in isolation.

The Attack

Latanya Sweeney (then a PhD student at MIT) purchased the Cambridge, Massachusetts voter registration roll for $20 — a publicly available government record containing name, address, ZIP code, date of birth, and sex for all registered voters. She performed a simple inner join on the three shared quasi-identifiers: {ZIP code, date of birth, sex}. In Cambridge specifically, 87% of residents were uniquely identified by just these three fields. She successfully identified the medical records — including diagnoses and prescriptions — of the Massachusetts Governor William Weld, and mailed them to his office to demonstrate the vulnerability.

Risks & Consequences

Medical records are among the most sensitive personal data: insurance discrimination, employer discrimination, social stigma, intimate partner surveillance. This attack predates HIPAA's full enforcement. Sweeney's paper ("k-anonymity: A model for protecting privacy," 2002) directly informed HIPAA Safe Harbor's 18-identifier list. Despite this, later studies showed that even HIPAA Safe Harbor-compliant data could be re-identified using richer auxiliary datasets — demonstrating that field suppression alone is insufficient.

Differential Privacy Mitigation

DP applied to the GIC data before publication would have injected Laplace noise into continuous fields such as age and charges, and would have suppressed or generalised rare {ZIP, age, sex} combinations below a minimum count threshold. An individual with a unique combination — a 52-year-old woman with a specific condition in a small ZIP code — would either not appear in the output or would appear with generalised age (e.g. "50–54") and a randomised charge value, breaking the exact join condition that made re-identification possible.

The DP lesson Sweeney's attack required no hacking — only a $20 public database and a laptop. The GIC data was "anonymised" by removing obvious identifiers, but retained enough quasi-identifiers to reconstruct identity via linkage. DP prevents this by guaranteeing that the presence or absence of any individual shifts the output distribution by at most eε — making exact-match linkage probabilistically impossible, not just procedurally discouraged.

The Epsilon & Cost Distinction — Secret Map Explained

In Module 6, two epsilon-like numbers appear simultaneously in the Peek Precision panel. This is one of the most common points of confusion for first-time players — but the distinction it illustrates is fundamental to how real DP systems work.

The confusion When the slider is at its minimum, the Peek Precision panel shows ε per peek: 0.10 and below it Next peek: ε0.24. These are not the same number — and they are not supposed to be.
ε per peek — 0.10

This is the precision setting you chose with the slider. It controls how much Laplace noise is applied to the peek — how sharp or blurry the revealed tile appears. Low ε (0.10) = lots of noise = very blurry tile. High ε (1.50) = little noise = crystal clear. This is the "signal quality knob."

Next peek: ε0.24 — the budget cost

This is the budget deduction that will be subtracted from your 2.0 ε wallet when you tap the tile. It is not identical to the precision value — it is calculated from a formula that includes a floor cost, because in a real DP system every query has a minimum cost.

The Cost Formula

-- BASE_COST = 0.30 (the minimum cost for any peek, even the blurriest) cost = BASE_COST × (0.7 + ε × 0.9) -- At minimum precision (ε = 0.10): cost = 0.30 × (0.7 + 0.10 × 0.9) = 0.30 × (0.7 + 0.09) = 0.30 × 0.79 = ε0.24 -- At maximum precision (ε = 1.50): cost = 0.30 × (0.7 + 1.50 × 0.9) = 0.30 × (0.7 + 1.35) = 0.30 × 2.05 = ε0.62

Why the Floor Cost Matters

In a real differential privacy system, querying a database is never free — even a query that returns heavily noised output still consumes epsilon from the privacy budget. This is because the act of answering any query reveals something about the dataset, even if the answer is buried under noise. A system that allowed zero-cost queries would let an attacker run unlimited queries, average away the noise across thousands of results, and reconstruct exact values — defeating differential privacy entirely.

The BASE_COST of ε0.30 in the Secret Map encodes this principle. Even your blurriest, cheapest peek (ε=0.10, cost=ε0.24) still depletes your wallet, because that blurry glimpse still incrementally narrows your uncertainty about where the treasure is. Eight minimum-cost peeks (8 × 0.24 = 1.92 ε) exhaust nearly all of your 2.0 ε budget — just as in a real DP system, a patient attacker running many cheap queries will eventually deplete the budget just as surely as running fewer expensive ones.

One-Line Summary

  • ε per peek = how sharp your view is (the noise level, the signal quality)
  • Next peek cost = how much budget that sharpness burns (always ≥ BASE_COST)

They move in the same direction — higher precision always costs more — but they live on different scales. This mirrors how real DP deployments report two numbers: the mechanism's epsilon parameter (precision) and the accumulated privacy loss (budget consumed). They are related but not identical.
